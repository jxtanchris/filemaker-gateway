 Context
  
 Part 2 已完成 Gateway 核心引擎，4 个 FileMaker Tools 为 stub。Part 3 目标：
 将 AI 能力无缝接入 FileMaker，完成从 FileMaker → Gateway → LLM → FM Data 的完整闭环。

 双向通信：
 - FM → Gateway: FileMaker 通过 Insert From URL 调用 Gateway（FileMaker 是客户端）
 - Gateway → FM: AI Agent 通过 FileMaker Data API 读写 FM 数据（Gateway 调用 FM Server）

 选型：FileMaker Data API（用户选择），不需要 proofkit-mcp 实时连接。

 Part 3 整体架构

 FileMaker Pro
    │ Insert From URL (POST /chat)
    ▼
 FileMaker AI Gateway (127.0.0.1:8080)
    │ AgentLoop → AgentRunner → DeepSeek
    │
    │ Tool 需要读数据时:
    ▼
 FileMaker Data API (FM Server)
    │ REST: /fmi/data/v1/databases/{db}/layouts/{layout}/...
    ▼
 FileMaker Server (业务数据库)

 两种调用方向

 方向 1: FM → Gateway (Insert From URL)
 FM Script "AI Chat"
   → Set Variable [$payload = JSONSetElement(...)]
   → Insert From URL [http://127.0.0.1:8080/chat, $payload]
   → JSON 解析 → 展示结果

 方向 2: Gateway → FM (Data API)
 AgentRunner 调用 filemaker_query tool
   → fm_client.py
   → POST https://fm-server/fmi/data/v1/databases/MyDB/layouts/Contacts/_find
   → 返回数据 → LLM 整理 → 回复用户

 Phase 1: Data API 客户端 (filemaker_gateway/fm/)

 新建 filemaker_gateway/fm/ 模块，封装 Data API 调用。

 filemaker_gateway/fm/
 ├── __init__.py
 ├── client.py       # FMDataClient: auth, CRUD, find, script

 FMDataClient 接口：
 class FMDataClient:
     def __init__(self, host, database, username, password):
         self.base_url = f"https://{host}/fmi/data/v1/databases/{database}"

     async def login() -> str                        # → token
     async def logout(token) -> None

     # CRUD
     async def get_records(layout, offset, limit, sort) -> list[dict]
     async def get_record(layout, record_id) -> dict
     async def create_record(layout, field_data, script?, script_param?) -> dict
     async def update_record(layout, record_id, field_data, mod_id?) -> dict
     async def delete_record(layout, record_id) -> None

     # Query
     async def find(layout, query, sort, offset, limit) -> list[dict]

     # Script
     async def run_script(layout, script_name, script_param?) -> dict

     # Metadata
     async def get_layouts() -> list[str]
     async def get_layout_metadata(layout) -> dict
     async def get_scripts() -> list[str]

     # Container (for OCR)
     async def upload_container(layout, record_id, field_name, file_path) -> dict
     async def get_container_url(layout, record_id, field_name) -> str

 Phase 2: 替换 FM Tool Stub（Gateway 侧）

 将 4 个 stub 替换为真实的 Data API 实现。

 修改文件：
 - tool/filemaker/query.py — execute() 调用 fm_client.find() / fm_client.get_records()
 - tool/filemaker/record.py — execute() 调用 fm_client.create/update/delete_record()
 - tool/filemaker/script.py — execute() 调用 fm_client.run_script()
 - tool/filemaker/layout.py — execute() 调用 fm_client.get_layouts()

 依赖注入： Tools 需要获取 FMDataClient 实例。两种方式：
 1. 简单方案：Tools 在 __init__ 时接收全局 fm_client（在 main.py 启动时注入）
 2. 优雅方案：Tool.execute() 通过 ToolRegistry 的 runtime context 获取

 采用简单方案：Tool 初始化时传入 client，ToolLoader 不支持参数的话就改为手动注册。

 实际改进方案： 修改 ToolLoader.load() 支持传入 keyword arguments 给 tool 构造函数。在 main.py 中：
 fm_client = FMDataClient(config.fm.host, ...)
 loader.load(registry, fm_client=fm_client)

 FileMaker 4 个 Tool 的构造函数接受 fm_client 参数：
 class FileMakerQueryTool(Tool):
     def __init__(self, fm_client: FMDataClient | None = None):
         self._fm = fm_client

     async def execute(self, action, layout=None, query=None, limit=100):
         if self._fm is None:
             return ToolResult.error("FM Data API not configured")
         if action == "select":
             records = await self._fm.get_records(layout, limit=limit)
             return json.dumps(records, ensure_ascii=False)
         elif action == "find":
             records = await self._fm.find(layout, json.loads(query))
             return json.dumps(records, ensure_ascii=False)

 Phase 4: OCR Tool 实现

 将 tool/ocr.py 从 stub 升级为真实实现。

 4.1 GLM OCR 集成

 如果用户已有 GLM OCR 实现：
 class OCRTool(Tool):
     def __init__(self, glm_api_key: str | None = None):
         self._glm_key = glm_api_key

     async def execute(self, image_url, ocr_type="glm"):
         if ocr_type == "glm":
             return await self._glm_ocr(image_url)
         elif ocr_type == "invoice":
             return await self._invoice_ocr(image_url)
         elif ocr_type == "pdf":
             return await self._pdf_ocr(image_url)

 4.2 Gateway 接收图片的方式

 Gateway POST /chat 支持 media 字段：
 {
   "session": "ocr-123",
   "message": "请识别这张发票",
   "media": ["data:image/png;base64,iVBORw0KG..."]
 }

 在 AgentLoop._build() 中，将 media 转为 LLM vision 格式：
 if ctx.media:
     content = [{"type": "text", "text": user_message}]
     for media_url in ctx.media:
         if media_url.startswith("data:image"):
             content.append({"type": "image_url", "image_url": {"url": media_url}})
     messages.append({"role": "user", "content": content})

 Phase 5: 端到端工作流与部署

 5.1 发票 OCR 完整工作流
 1. FM 用户拍照/导入发票到容器字段
 2. FM 脚本 "AI OCR 发票识别" 运行
 3. Base64Encode → Insert From URL → Gateway /chat
 4. AgentLoop 收到消息 + 图片
 5. LLM 判断需要调用 ocr tool
 6. AgentRunner 调用 OCRTool
 7. OCR 返回结构化 JSON (发票号/日期/金额/抬头)
 8. LLM 整理回复
 9. Gateway 返回 answer 给 FM
 10. FM 脚本 JSON 解析 → 填入对应字段

 5.2 AI 查询业务数据工作流
 1. FM 用户在输入框输入 "查最近30天的订单，金额大于10000的"
 2. FM 脚本 POST /chat
 3. LLM 理解意图 → 调用 filemaker_query tool
 4. Gateway 通过 Data API 执行 find
 5. FM Server 返回订单列表
 6. LLM 整理为易读格式
 7. Gateway 返回 answer 给 FM
 8. FM 展示在结果字段或 Web Viewer 中

 2. Phase 2 — 替换 4 个 FM Tool stub → 真实实现
 3. Phase 3 — FM 脚本模板（可粘贴到 FM Script Workspace）

 验证方式

 每个 Phase:
 - 单元测试（mock Data API 响应）
 - Phase 1-2: 手动用 httpx 测试 Data API 连接
 - Phase 3-4: 用 curl 模拟 FM Insert From URL 请求，传 media 测试 OCR
 - Phase 5: 完整工作流测试（如果 FM Server 可用）

 与 Part 2 的兼容性

 - fm_data_api.enabled = false 时，FM Tools 自动降级为 stub 模式，Gateway 不受影响
 - 所有现有 API 端点不变
 - 测试不依赖真实 FM Server
